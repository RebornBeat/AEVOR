## [Unreleased]

### Milestone 39 — Balance settlement + rule enforcement + economics-in-throughput + economic sweet spot (2026-07-21) ✅ DONE
Closed the settlement gap flagged in M38 (fees were computed/reported but never debited), made the finalized economics **binding rules every node must follow**, and ran the economics **inside** the throughput pipeline (not a standalone simulation) to finalize the economic sweet spot against the measured throughput sweet spot. Analysis in `21_BALANCE_SETTLEMENT_AND_ECONOMIC_SWEET_SPOT.md`.
- **Per-transaction settlement.** The executor now exposes per-tx execution gas (`ProgramOutcome.accepted_tx_gas`, keyed by tx hash). The engine holds an account ledger (`Address -> Amount`); two shared helpers run on **both** production paths (`process_block` and PoU `produce_attested_batch`): `affordability_filter` (the account-level **abuse guard** — reserves each sender's intrinsic fee, drops those who can't cover it, before execution; feeless subnets admit all) and `settle_and_advance` (debits each accepted sender `(intrinsic+exec)·base_fee`, credits the validator, advances the congestion market). **Conservation by construction:** `fee_charged == Σ sender debits == validator reward`.
- **`fund()` gated to genesis.** Permitted only at `height == 0`, rejected afterward — the test/benchmark funding path **cannot mint on a running chain**. Genesis allocation is agreed as part of the genesis block; thereafter balance moves only via settlement (and future transfers).
- **Rule-version enforcement.** `PROTOCOL_RULES_VERSION` is folded into every `ExecutionAttestation` body, so a validator running different economics signs a different body and its blocks fail verification on nodes at another version. This is what forces every node onto the same finalized code — divergence is cryptographically rejected. Combined with the already-attested `tx_commitment` (which txs ran) and deterministic settlement, balances are a deterministic function of attested inputs and cannot diverge undetected.
- **Economics hooked into the throughput benchmarks.** `bench_full_pipeline`, `bench_combined_pou_scaling`, and `bench_fee_market_dynamics` now settle fees inline and report gas/fee/reward alongside tx/s (benchmark openers `bench_fund` the shared zero sender; kept on a fee subnet so economics actually accrue). Re-ran the suite with the parallel additions (M31–M34) in place: **production ~12 500 tx/s (batch ≈ 2 000), verify-attest ~1.2 M tx/s (~98×), aggregate projection 128 lanes → 1.56 M, 10 000 → 122 M tx/s** (1M–100M+ confirmed as projection). Congestion dynamics reproduce the finalized M38 market exactly (base fee 1000→1634→…→962; PQ bloat ~11×; reward 13 730 350 nano) but now **actually debited**.
- **Economic sweet spot = throughput sweet spot.** fee/tx is flat at **0.000485 AVR** at every batch size; the congestion target (~15 M gas ≈ 30 900 tx/block) sits far above the per-lane throughput sweet spot, so at the sweet spot the base fee stays at the floor and fees are cheapest — congestion pricing engages only *past* the sweet spot. Fees as cheap as possible, rising only to hold off genuine congestion.
- **Tests.** New e2e `balance_settlement_debits_senders_credits_validator_and_guards_abuse` (conservation + abuse guard + feeless) and `independent_nodes_settle_identically_same_rules_same_result` (two nodes → identical balances/fee/reward/root/next-base-fee; no minting post-genesis).
- **Balances committed and synced everywhere — cheaply, not in the Merkle root.** Per-account balance changes now ride in the block `StateDelta` and are bound by a `balance_commitment` (one BLAKE3 over the changes) folded into the attestation body. `apply_attested_batch`/`apply_lane_round` check the shipped balances against the commitment (tamper-evident) and apply them to the ledger — **HashMap writes, no Merkle** — so a validator that only *verified* a block ends with the producer's balances and can *produce* correctly later (`verifier_stays_balance_consistent_on_fast_path`). A producer that ships/settles wrong balances is caught (hash mismatch at apply, or a re-executing validator recomputing the commitment → slashing); no equivocation (the commitment is signed). `bench_balance_in_root_overhead` shows committing balances into the **Merkle root** would cost **+112% to +136%** on the verify path (roughly halving it); the `balance_commitment`+`StateDelta` path adds ~nothing — the verify sweep is unchanged at ~1.0–1.15M tx/s. Light-client balance proofs remain an opt-in capability at the Merkle cost.
- **Multi-lane balance settlement decided + validated: sender-sharded routing.** For the fee-only economy, concurrent lanes touch disjoint accounts (each sender's txs in one lane; each lane credits its own validator), so `apply_lane_round`'s per-lane absolute balance deltas settle correctly with no cross-lane contention or double-spend. Proven by `multi_lane_settlement_correct_under_sender_sharding` (two lanes, two senders, both debits reflected). Transfers (future) get a cross-shard protocol. Design capture: `22_TEE_CORRUPTION_DETECTION_AND_SCALING.md`.
- **Corruption detection: reconsidered against real TEE and captured (not committed).** A re-execution corruption primitive was prototyped, measured (runs at the ~12k tx/s produce rate, ~100× slower than fast-path verify — anti-scaling), found redundant with real TEE (whose attestation proves correct execution instantly), and **reverted**. The production mechanism is TEE attestation verification (enclave code-measurement registry + signature, O(1)/block), built with F-E1; an optional tunable sampled re-execution quorum is a defense-in-depth backstop against TEE compromise. Full design in doc 22.
- **Transport seam + in-process multi-node round (network logic, single-core).** New `node/src/transport`: a `NetworkMessage` set + `Transport` trait + an in-memory backend connecting N `NodeEngine`s in one process. The full macro-DAG round runs over it — `multi_node_round_over_transport_converges_with_settlement`: validators produce lanes on disjoint accounts, broadcast, apply the round, and **converge to one state root with consistent settled balances**. Validates the network *logic* (lane exchange + `apply_lane_round` + balance-delta settlement + convergence) single-core; the real gossip/TCP backend (`aevor-network`, 61 tests) plugs in behind the same trait and needs multi-machine to prove. Confirmed no throughput degradation from any M39 work: economics byte-identical across benchmark runs (fee/tx 485,000; base fee 988–1007; reward = debit), wall-clock variation is single-core sandbox noise (produce + verify move proportionally).
- **Attack surface tested + a double-spend gap fixed.** Found during review: `apply_lane_round`'s cross-lane conflict check only rejected duplicate tx-commitments, **not** two lanes writing the same object — a real cross-lane double-spend (last-write-wins would silently drop one). Fixed (step 2b rejects same-object across lanes; step 2c rejects same-account across lanes for balance conflicts). Multi-lane tests updated to realistic distinct senders. New attack tests, all passing: `cross_lane_object_double_spend_is_rejected`, `cross_lane_same_account_settlement_is_rejected`, `duplicate_transaction_set_across_lanes_is_rejected`, `lane_not_forking_from_round_base_is_rejected`, `tampered_lane_balance_delta_is_rejected` — alongside existing coverage for within-lane conflicts, forged/tampered attestations + deltas, balance overdraft, post-genesis mint, and bad signatures (doc 22 §10). The new checks are O(objects+accounts) HashSet passes per round (multi-lane path only); benchmarks re-run show no degradation — economics byte-identical, single-lane throughput within variance, multi-lane apply ~402k–810k tx/s across 1–32 lanes.
- **Throughput re-verified by repeated runs + full scaling-factor analysis (doc 23).** Ran `bench_full_pipeline` 3× and `bench_multi_lane_round` 2× back-to-back: timing-dependent numbers wander ±6–8% (up to ~2× at 32 lanes) on identical code while deterministic outputs (base fee 990, gas 2,425,000, fee/tx 485,000, reward) are byte-identical every run — variance, not degradation; verify back at ~1.0–1.1M. Clarified the model: throughput ≈ N_lanes × per-lane production; production (~11k/lane/core) is the bottleneck, verification has ~100× headroom (sharded past the ~100-lane crossover). The single-vs-multi-lane "gap" was measurement-shape + variance; the per-node decline with lane count is one verifier doing N× work with growing state — exactly what sharded verification replaces. Inventoried all captured scaling factors (per-node multi-core, multi-lane production, sharded verification, PoU headroom, off-Merkle balances, O(1) finality, network availability, accelerator framework) and flagged two extreme-scale factors as deliberate future work: **state sharding** (partitioned state ownership) and **cross-round pipelining**.
- **Controlled experiment: multi-lane scaling is real, not CPU-maxing (doc 23 §7–8).** New `bench_controlled_lane_independence` isolates the work from contention — 32 disjoint lanes on fresh engines, measuring per-lane production cost vs lane index. Across 3 runs, per-lane cost is FLAT (first-half vs second-half ratio 1.01/1.00/0.99), mean stable at ~13.4k tx/s (±0.5% run-to-run), min..max spread is CPU noise. Flat per-lane cost + disjointness ⇒ N cores give N × per-lane linearly (96→1.28M, 9000→120M). The decline in `bench_multi_lane_round` is one verifier accumulating all N lanes' state (full verification on one core), not per-lane degradation. `bench_sharded_verification_scaling` (fixed to use distinct senders — the double-spend check had correctly caught its ZERO-sender violation) confirms the verification side: validator-0's slice stays bounded (2–5 lanes) as N grows 8→128. Full spectrum verified: per-lane production (flat) → ~96-lane crossover → sharded (bounded slice, measured) → ~100M+ aggregate projection at ~9000 lanes, now backed by measured independence + measured slice-bounding rather than arithmetic alone. No structural change from M31/M32 lane model.
- **Live network wire: the macro-DAG round over a real transport (doc 24).** Wired the full round to `aevor-network`'s transport. New `node::network::run_round` (generic over `MessageTransport`): record round base → produce this validator's lane → broadcast → poll-collect peers' lanes to quorum → apply and converge. New `NodeEngine::apply_foreign_lanes`: a validator that already applied its own lane applies the OTHER lanes on top, fully verified + conflict-checked (double-spend defenses) + settled; disjoint lanes ⇒ identical root on every validator. `StateDelta`/`LaneBlock` are now `serde`; lanes cross the wire as bincode in `NetworkMessage::Block`. Validated single-core: `lane_wire_round_trip`, `apply_foreign_lanes_producer_flow_converges` (producer reaches same root + balances + ordering as a fresh verifier), and `run_round_over_real_transport_converges` (3 validators, each on its own thread, run `run_round` concurrently over a shared `LocalNetwork` and all converge to one root — the same `MessageTransport` trait the real `TcpTransport` implements). On hardware: bind `TcpTransport` instead of `LocalNetwork`, same `run_round`, no code change. Real gossip already exists in `aevor-network` (`TcpTransport`: length-prefixed framing, background accept loop). Production round coordination (round numbers, quorum certs, view changes, equivocation-over-wire) layers on top of this exchange primitive.
- **Real TEE attestation (AWS Nitro) — production-ready, buildable, hardware-tested (doc 25).** PoU is now real, not simulated. Producer `aevor_tee::nitro::attest` issues the genuine NSM attestation request (`aws-nitro-enclaves-nsm-api`), binding the `ExecutionAttestation` body as `user_data`. Verifier `aevor_tee::nitro::verify::verify_document` (pure Rust — `ciborium`/`p384`/`x509-parser`/`sha2`) decodes the `COSE_Sign1`, validates the cert chain to the pinned AWS Nitro root (fingerprint `64:1A:…:5B`), and verifies the ES384 signature; `MeasurementRegistry`/`check_policy` enforce body-binding, freshness, and accepted PCR0/1/2(/8) — the code-identity/corruption-detection step (O(1), no re-execution). Wired into consensus: `ExecutionAttestation` carries an optional real `tee_document`; `seal` binds it on hardware (guarded by a `/dev/nsm` stat, sim fallback off-hardware) and `verify` checks it cryptographically when present, so `apply_lane_round`/`apply_foreign_lanes` verify real attestations unchanged. `check_measurement(registry, …)` enforces which enclave image may produce. 13 aevor-tee nitro tests (malformed rejected, registry/PCR8/policy). Builds + runs off-hardware via the sim fallback; producer runs on Nitro. Other 4 platforms follow the same shape.
- **Transport made production-first.** Corrected the framing: `ValidatorNetwork::bind` (real `TcpTransport`) is now the default production entry point for a live validator; `LocalNetwork` is explicitly the test-only injection. `run_round` stays the generic engine underneath. (Previously documented backwards — LocalNetwork as the norm.)
- **No degradation:** benchmarks re-run 2× after TEE + network changes — PRODUCE ~11.7–11.9k, VERIFY ~1.0–1.1M, economics byte-identical (base fee 990, fee/tx 485,000); the `is_available` guard keeps `seal` to one stat off-hardware.
- **State sharding (opt-in mode) + cross-round pipelining, merged not replacing (doc 26).** *State sharding* is a new `ShardingMode` on the engine — `Monolithic` (default; full state, unchanged production) or `Sharded { shard_id, assignment }` (stores/applies only owned objects/accounts), interchanged via `set_sharding`. `ShardAssignment` maps object/account ids to shards deterministically; the round-apply paths route storage through `apply_owned_delta` (owned-only), so two validators in different shards hold disjoint state covering the whole round (`sharded_mode_partitions_state_across_validators`) while the cross-lane double-spend defenses still run over the entire round. `CrossShard` classifies single- vs cross-shard writes; cross-shard atomicity reuses the per-lane attestation set. Remaining for full sharded operation: shard-aware execution on the producer + the live cross-shard driver (apply-side partition is done). *Cross-round pipelining* (`node/src/pipeline`, applies to BOTH modes) overlaps a round's O(1) finality with the next round's production: `RoundPipeline` advances applied state at production rate and trails finality certificates by the pipeline depth (`finality_trails_application_by_depth`, `drain_finalizes_remaining_in_order`). No degradation: `bench_full_pipeline` ×2 PRODUCE 11,830/11,495, VERIFY ~1.1M, economics identical; multi-lane apply on baseline (the ownership check is a no-op match in monolithic mode).
- **Shard-aware execution — sharded mode is now end-to-end (doc 26).** A sharded producer no longer executes everything. `CrossShard::coordinator` designates the shard owning a transaction's lowest-ordered written object as its executor — derived purely from the transaction, so every validator agrees with zero coordination. `ShardingMode::is_responsible_for` + the engine's `shard_admission_filter` run ahead of both production paths, so a sharded validator executes only what it coordinates; the rest are counted in `BlockOutcome.routed_to_other_shard` for routing, not dropped. `verify_execute_commit` stores only owned objects while the shipped `StateDelta` still carries every write — that combination *is* the cross-shard mechanism (one executor, one attestation, per-shard application; atomic because a verifier takes the whole attested delta or none). Tests: `shard_aware_execution_partitions_work_exactly_once` (4 shards × 24 txs: each shard executes a strict subset, `accepted + routed == total` per shard, executed counts sum to exactly the total — **each transaction executed exactly once**), `cross_shard_delta_ships_all_writes_while_storing_only_owned`, `exactly_one_shard_is_responsible_for_any_transaction`, `coordinator_is_the_shard_of_the_lowest_written_object`. Note: a sharded validator's Merkle root is a per-shard root (monolithic unaffected).
- **All five TEE platforms real and production-first (doc 27).** Beyond Nitro: **Intel SGX** (DCAP v3 quote — ECDSA P-256 over header‖report_body, QE report binds the attestation key via SHA-256, PCK chain to a pinned Intel root), **AMD SEV-SNP** (1184-byte report — ECDSA P-384 over bytes [0,672) with the VCEK key, VCEK chained to AMD root), **ARM TrustZone** (PSA attestation token — ES256 `COSE_Sign1` + PSA/EAT claims → per-component measurements), **RISC-V Keystone** (1352-byte report — two-link Ed25519 chain: device key signs SM report, SM key signs enclave report). Real formats/offsets/encodings throughout. Two new layers keep consensus platform-agnostic: `aevor_tee::registry` (`VerifiedEnclave`, `CodeRegistry`, `check_policy` — the network-agreed code identity across all platforms) and `aevor_tee::evidence` (`AttestationEvidence` enum + `produce()` using whichever TEE is present + `verify(&TeeTrustRoots)` dispatch; trust roots are network config, chip-specific VCEK travels with the evidence). Producer device interfaces: NSM ioctl (Nitro), `/dev/attestation/*` (SGX), configfs-tsm (SEV-SNP), `/dev/keystone_enclave`, OP-TEE `/dev/tee0`. `ExecutionAttestation.tee_evidence` replaces the Nitro-specific field; `verify` fails closed for platforms whose root is configuration; `check_measurement(registry, roots, …)` applies full policy. **Security fix found by testing:** the Keystone verifier's Ed25519 `verify` accepted an all-zero report (identity public key + zero signature satisfies the equation — small-order key attack); switched to `verify_strict`.

### Milestone 38 — Congestion-based fee market + validator rewards + PQ/privacy pricing + flexible subnet fees + tokenomics finalized (2026-07-18) ✅ DONE
Finalized the economic core as a *smart* congestion-driven mechanism, with full e2e simulation of congestion, post-quantum bloat, token price, and rewards. Reframed the Tokenomics reference for a **utility** chain (not funding-first).
- **One fee formula everywhere:** `fee = total_gas × base_fee`, matching the VM `GasMeter`. `total_gas = execution_gas (VM, incl. TEE premium) + intrinsic_gas`. New in `aevor-core::economics`: `intrinsic_gas(size, base, per_byte)` (flat base + per-byte charge — prices **post-quantum bloat**, ML-DSA ~5.5 KB ≫ Ed25519), `next_base_fee(base, gas_used, target, adjustment_bps, min)` (EIP-1559-style integer congestion rule), `FeePolicy::after_block(...)` (advances a `MarketBased` base fee; `Free`/`Fixed` unchanged), and constants `DEFAULT_INTRINSIC_TX_GAS=100`, `DEFAULT_GAS_PER_BYTE=1`.
- **Privacy & PQ priced through gas, not as bolt-on surcharges:** privacy runs in a TEE so it consumes more *execution* gas automatically (the existing TEE premium); PQ pays more *intrinsic* gas via the per-byte charge. Both also push the congestion controller, which is the correct treatment since both add real load.
- **Congestion, not token price:** the base fee rises when a block exceeds target utilization and falls toward the floor when idle; the protocol never reads a token price. Token price affects only the fiat display — the native fee is identical at any price (asserted in the simulation).
- **Validator rewards from usage:** the engine now tracks `fee_market: FeePolicy` and `validator_reward: Amount`; `process_block` charges the dynamic fee, credits it to the validator as the block reward, and advances the market for the next block. Usage funds security — no inflation. Sits on the same ledger as graduated slashing (M35). New engine getters `current_base_fee()`, `fee_market()`, `validator_reward()`.
- **Flexible subnet fees:** a subnet chooses its economics as one `FeeConfig` — `with_fee_config(...)` (share mainnet's model or bring its own), `flat_fee(price, ...)` (fixed price, `adjustment_bps=0`), `public_with_congestion(...)` (full tuning), or `feeless_permissioned(...)`/`FeeConfig::feeless()` (none). Sharing `FeeConfig::default()` inherits future mainnet changes automatically.
- **End-to-end simulation:** `fee_market_simulation_congestion_pq_price_rewards` (e2e) proves congestion raises then idle lowers the base fee, a PQ tx costs more than Ed25519, the validator accrues fees, and the native fee is token-price-independent. `bench_fee_market_dynamics` prints the base-fee trajectory (1000→1634→…→962), the ~11× PQ premium, cumulative reward, and fiat cost at $0.01–$150/AVR. `bench_gas_and_fee_estimates` updated to bloat-inclusive gas (simple tx 485 gas = 0.000485 AVR; heavy 3000-instr = 0.024461 AVR; feeless = 0).
- **Tokenomics finalized:** `Tokenomics.md` gained a status banner (utility-first; fees not inflation fund validators; feeless subnets supported; the finalized section governs over the prose) and a concrete **"Finalized Fee-Market and Reward Mechanism"** section (the formula, congestion rule, rewards, flexible subnet modes, measured numbers). Analysis in `20_FEE_MARKET_ALGORITHM_AND_TOKENOMICS.md`.
- **Verified:** `aevor-core` economics 26 tests + clippy 0; node 56 lib + 22 e2e + clippy (lib + tests) 0.
- **Honest caveat:** fees/rewards are computed, accrued, and reported (`BlockOutcome.fee_charged`, `validator_reward`) but **not yet debited from payer `Balance`s** — the settlement layer exists in `aevor-core` and is the next economic step.

### Milestone 37 — Fee model unified to single-source-of-truth + gas/fee estimates + subnet/PoU cleanliness review (2026-07-18) ✅ DONE
Addressed a real divergence flagged in review: the first `SubnetPolicy` invented its own `gas_price_nano`, disconnected from the canonical `FeeConfig` and the VM's `GasMeter`. Reconciled so subnets branch cleanly off the one mainnet flow and inherit any mainnet fee change.
- **Confirmed one canonical flow.** `new_simple` is a convenience *builder* over the single `Transaction` struct (used by `aevor-client` too), not a second tx type. `open_on_subnet` delegates to the same `open_with_backend` + same `process_block → verify_execute_commit → process_program_batch` pipeline (same signature gate, micro-DAG rejection, VM, PoU attestation, Merkle commitment). The subnet branches at exactly three points — fee, admission, privacy baseline — and nowhere else. Mainnet is `SubnetPolicy::public_mainnet()`.
- **Fees are now single-source-of-truth.** `SubnetPolicy` **carries the canonical `FeeConfig`** (the same struct mainnet uses); `fee_for(gas_used)` is the **same formula** as the VM `GasMeter::fee` — `fee = gas_used * gas_price` — with the price sourced from that `FeeConfig` (feeless ⇒ 0). Added `effective_gas_price()`, `feeless()`, and `fee_for_tx(gas_used, max_gas_price)` which honors the transaction's own gas-price cap. `public_mainnet` = `FeeConfig::default()` (1000 nano/gas); `feeless_permissioned` = `FeeConfig::feeless()`; `from_deployment_config` maps `fees_enabled` to the shared or feeless `FeeConfig`. Any future mainnet fee change now propagates to every subnet with no code change.
- **Measured gas/fee estimates** (`bench_gas_and_fee_estimates`, values assert against `BlockOutcome.fee_charged`): gas is instruction-based (~1.67 gas/opcode). A simple tx ≈ **5 gas → 5,000 nanoAVR = 0.000005 AVR** at mainnet price; a 3,000-instruction contract ≈ 0.005 AVR; feeless subnet = **0**. Gas already includes the VM schedule's TEE execution premium, so these are PoU-inclusive costs.
- **Cross-chain comparison + PoU/sweet-spot review** in `19_FEE_GAS_ANALYSIS_AND_SUBNET_UNIFICATION.md`: AEVOR's 0.000005 AVR/tx matches Solana's native-unit base-fee magnitude (0.000005 SOL) and sits far under Ethereum L1 in any plausible fiat mapping; feeless subnets are $0. PoU scaling is subnet-independent (attestation is over state, not fees), so the ~1k–10k verify-attest sweet spot and `aggregate = lanes × per-lane` hold **per subnet**, composing across two horizontal axes (lanes, subnets). Honest caveats: no definitive USD/tx without a token price; fees are computed/reported, not yet debited from a balance model.
- **Verified:** workspace builds; node clippy (lib + tests) **0**; 21 e2e + 54 node lib tests green (9 subnet unit tests, incl. new mainnet-price/per-tx-cap/from-config-fee tests).

### Milestone 36 — F-C1 subnet economics + enforcement end-to-end (feeless/fee · permissioned/permissionless · privacy baseline) (2026-07-18) ✅ DONE
Finalizes the subnet model: a subnet launches with a policy it enforces on every transaction and object, exactly as an individual object carries its own privacy level. Reviewed against the whitepaper (feeless permissioned-subnet economics, permissioned subnets with custom privacy policies, multi-subnet coordination with different baselines) and wired end-to-end.
- **New `node::subnet::SubnetPolicy`** — three independent axes as typed values: **feeless vs fee** (`fee_for(gas_used)` → zero on a feeless subnet, else `gas_used * gas_price`), **permissioned vs permissionless** (`admits(sender)`), and an **enforced privacy baseline** (`allows_privacy(level)`, with `Public < Protected < Private` ordered). Constructors `public_mainnet` / `feeless_permissioned` / `fee_public`, plus `from_deployment_config` bridging the declarative `aevor-config` `SubnetDeploymentConfig` (the operator's source of truth) into the enforced policy.
- **Feeless economics enforced end-to-end.** `BlockOutcome` now records `fee_charged: Amount`, computed from the block's gas via the subnet policy. Gas is still metered (it bounds work); the *fee* on that gas is zero on a feeless subnet. e2e `feeless_subnet_charges_no_fee_fee_subnet_charges_gas`: the same batch pays 0 on a feeless subnet and `gas_used * 2` on a 2-nano/gas subnet.
- **Privacy baseline enforced at dApp deployment.** New `NodeEngine::open_on_subnet(dir, owner, subnet, dapp_privacy, security)`: a dApp whose privacy is **below** the subnet baseline is **rejected outright** (privacy is architecturally enforced, never downgraded); at/above the baseline is accepted and objects are stamped at the dApp's level. e2e `subnet_privacy_baseline_rejects_below_and_stamps_at_level`.
- **Permissioned admission enforced at submit.** A permissioned subnet rejects a non-permitted sender before it enters the mempool; a permissionless subnet admits everyone. e2e `permissioned_subnet_admits_only_permitted_senders` (senders set before signing so `signing_bytes` commits to them).
- **Doc:** `18_SUBNET_MODEL_AND_DAPP_DEPLOYMENT.md` — full whitepaper review of the subnet model and how dApps deploy with settings, axis by axis, with the enforcement mapping.
- **Verified:** workspace builds; node clippy (lib + tests) **0**; 21 e2e + 50 node lib tests green (incl. 5 new subnet unit tests + 3 new subnet e2e tests).

### Milestone 35 — F-A3 corruption→slashing loop + F-B1 stub SNARK verifiers gated fail-closed (2026-07-18) ✅ DONE
Two finalization items: close the "valid until proven corrupted" economic loop, and make sure no stub can pose as verification.

### Milestone 21 — Performance: parallel execution + O(n) conflict detection + incremental Merkle (2026-07-18) ✅ DONE
The performance milestone from the canonical design review. The benchmark's superlinear slowdown is **gone** — throughput is now flat with batch size — and the win is *measured*, not asserted.
- **Parallel execution (micro-DAG wired into the executor).** The executor ran the accepted transactions sequentially. It now runs in three passes: (1) sequential, deterministic **conflict rejection** → the independent set; (2) that independent set's VM programs **execute in parallel** (rayon) — they have disjoint read/write sets by construction, so execution is order-independent and the result is deterministic; (3) sequential, in-order **apply** (persist writes / record failures). This is the micro-DAG's independent set actually running in parallel.
- **O(n) conflict detection (was O(n²)).** The pairwise scan (each tx vs every prior accepted tx) is replaced by two `HashSet`s (accepted writes / reads); a tx conflicts iff any of its writes hits an accepted write/read or any of its reads hits an accepted write — the *exact* `ConflictDetector` relation, aggregated. This was the main source of the superlinear slowdown on large batches.
- **Incremental Merkle.** `MerkleProver` now (a) keeps a **precomputed leaf hash per key** so a rebuild never re-hashes unchanged values (O(1) amortized per write vs O(n) per root), and (b) **caches the root**, invalidated on mutation, so the several `root()`/`prove()` calls the engine makes per block don't each rebuild the tree.
- **Measured impact (debug build, single-node, in-process):**
  - Execution throughput: **flat ~14k tx/s across 1k→25k txs** (was 13.4k → **3.8k**, degrading). At 25k that's a **~3.8× improvement**, and the superlinear shape is eliminated.
  - Merkle proof generation at 25k objects: **~20 ms → ~5.1 ms** (~4×), from the cached leaf hashes.
  - Finality-proof aggregation unchanged and still fast (0.13 ms at 4 validators → 4.3 ms at 128).
  - **Determinism preserved:** all 12 end-to-end tests pass (conflict rejection, two-node convergence, multi-block, etc.).
